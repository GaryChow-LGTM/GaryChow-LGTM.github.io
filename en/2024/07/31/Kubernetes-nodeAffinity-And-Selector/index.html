Kubernetes nodeAffinity & nodeSelector Intros

Affinity-based Scheduling in Kubernetes

K8s POD is automatically scheduled to a NODE. By default, the scheduler considers whether resources are sufficient and tries to keep the load as evenly distributed as possible. However, sometimes we need finer-grained control over POD scheduling, for example, controlling whether PODs run on the same NODE. This is where a Kubernetes concept comes in: affinity. Affinity is mainly divided into two categories: nodeAffinity and podAffinity.

nodeSelector

Users can flexibly leverage label to manage resources in a cluster. The most common example is that a service selects PODs by matching label. Likewise, POD scheduling can also be targeted to specific deployments based on node labels.

nodeAffinity

nodeAffinity refers to node affinity, with its counterpart being Anti-Affinity. This approach is more flexible than the nodeSelector above, allowing for some simple logical combinations rather than just simple equality matching. Scheduling can be divided into soft and hard policies. A soft policy means that if there is no node that satisfies your scheduling requirements, the POD will ignore that rule and continue the scheduling process. In other words, it’s the “it’s best if it’s satisfied, but it doesn’t really matter if not“ policy. A hard policy is much stricter: if no node satisfies the conditions, it will keep retrying until the conditions are met. In short, it’s the “you must meet my requirements, otherwise I won’t do it“ policy.

nodeAffinity is divided into the following two policies:

  1. preferredDuringSchedulingIgnoredDuringExecution — soft policy
  2. requiredDuringSchedulingIgnoredDuringExecution — hard policy.

The above POD first requires that the POD cannot run on nodes 140 and 161, and if there is a node that satisfies source=qikqiak, it will be preferentially scheduled onto that node. Likewise, we can use the descirbe command to check whether the specific scheduling result meets our expectations. The matching logic here is that a label’s value is within a given list. The operators currently provided by Kubernetes are as follows:

  • In: the label’s value is in a given list
  • NotIn: the label’s value is not in a given list
  • Gt: the label’s value is greater than some value
  • Lt: the label’s value is less than some value
  • Exists: a label exists
  • DoesNotExist: a label does not exist

If there are multiple options under nodeSelectorTerms, satisfying any one of the conditions is enough; if there are multiple options in matchExpressions, all of these conditions must be satisfied simultaneously for the POD to be scheduled normally.

podAffinity

Both of the approaches above let a POD select a node. Sometimes we also want to perform scheduling based on relationships between PODs. The podAffinity concept introduced in Kubernetes 1.4 can fulfill this need.

Similar to nodeAffinity, podAffinity also has two scheduling policies: requiredDuringSchedulingIgnoredDuringExecution and preferredDuringSchedulingIgnoredDuringExecution. The only difference is that if we want to use mutual exclusion, we need to use the podAntiAffinity field. Taints and Tolerations

For nodeAffinity, whether hard or soft policy, the goal is to schedule the POD onto the expected node. Taints does the exact opposite: if a node is marked with a Taint, unless the POD is also marked as able to tolerate tainted nodes, no PODs will be scheduled to that tainted node.

For example, if users want to reserve the Master node for Kubernetes system components, or reserve a group of nodes with special resources for certain PODs, taints become very useful—PODs will no longer be scheduled onto nodes marked with a taint. Here is an example of tainting a node:

There are three available options for effect, which can be set according to actual needs:

  1. NoSchedule: Unless there is a matching toleration specification, new Pods will not be scheduled onto the tainted node. Pods currently running on the node will not be evicted.
  2. PreferNoSchedule: PreferNoSchedule is the “preferred” or “soft” version of NoSchedule. The control plane will try to avoid scheduling Pods that cannot tolerate the taint onto the node, but this is not guaranteed to be completely avoided.
  3. NoExecute: This option means that once the Taint takes effect, any POD running on the node that does not have a matching Tolerate setting will be evicted directly.
