Spark HistoryServer Intros and Troubleshooting

HistoryServer Introduction

HistoryServer is an HTTP service built into Spark, we can start it with the following code. When HistoryServer is started, it starts two threads to parse and clean the log files.

This creates a web interface at http://:8080, listing incomplete and completed applications and attempts.

In order to better introduce HistoryServer, the flowchart of HS is listed here.

Spark History is based on the Spark Event (Spark Event) system. During the Spark task, a large number of SparkListenerEvents containing running information are generated, such as ApplicationStart/StageCompleted/MetricsUpdate, etc. Each has a corresponding SparkListenerEvent implementation. All events are sent to the ListenerBus and listened to by all listeners registered in the ListenerBus. EventLoggingListener is a listener that generates event logs. It serializes the event into an event log file in Json format and writes it to a file system (such as HDFS). Usually the files of a machine room’s tasks are stored in a single path.

On the History Server side, the core logic is in the FsHistoryProvider. The FsHistoryProvider maintains a thread that periodically scans the configured event log storage path, traverses the event log files in the path, extract the profile information (mainly Appliaction_id, user, status, start_time, end_time, event_log_path) and maintain a list. When the user accesses the UI, the task required by the request will be searched from the list, and if it exists, the corresponding Event log file will be read completely for parsing.

Issue

In some cases, after the user submits the job using Spark, the log information corresponding to appId cannot be found in Spark HistoryServer WebUI(v3.2.1).

As a result, the user cannot analyze the job log through HS. In view of this problem of source code for reading and analysis, finally found that the problem occurs in the use of a data structure caused by multithreading.

Log Parsing Process

The HS runs the checkForLogs to build the application list based on the current contents of the log directory for every 10 seconds by default. In each turn of execution, this method scans the specified logDir and parse the log files to update its KVStore Listing, include two method:

  1. update/add log file records respectively
  2. remove stale log file records.

These two methods are committed by submitLogProcessTask to different threads for execution

Issue Scenario

When checkForLogs is executed for the first time, the ABC job submitted by the user is being executed. The file name in the log directory is ABC.inprocess, so the KVstrore Listing adds two new messages, such as:

In the next run of checkForLogs, now the ABC application has finished, the log file /tmp/ABC.inprocess has been deleted, and the new log file path is /tmp/ABC, so in this run of checkForLogs, two threads will start and do following things:

Thread 1 will update/add log file records, this thread will parsing the log path /tmp/ABC. First it will update the <'ABC', InfoData<'ABC'>> in the KVstore Listing, second it will add a new record <'/tmp/ABC', InfoData<'/tmp/ABC'>> in KVstore Listing.

Thread 2 will remove stale log file records, this thread will notice that the log path /tmp/ABC.inprocess has disappeared. First it will filter this path as a stale data, second it will start cleanAppData method, in this method the following code will be executed:

In most cases, when this method load the appId, Thread 1 already finished updating the <'ABC', InfoData<'ABC'>> in the KVstore Listing, so when you next execute the following code, the value of isStale will be false.

But in some cases, when this method load the appId, Thread 1 does not finish the update, so when you next execute the code above, the value of isStale will be true, and the worst thing is it will delete the log file record <'ABC', InfoData<'ABC'>> in KVstrore Listing after Thread 1 has finished update it.

So the log file record <'ABC', InfoData<'ABC'>> will removed forever the when users access the WebUI for this application, and Application not found is shown up while the log for the app does exist.

One thing to note the log file record <'ABC', InfoData<'ABC'>> will not add in the KVstore Listing again, because in the next of checkForLogs, <'tmp/ABC', infoData<'tmp/ABC'>> indicates that the log file of ABC does not need to reload, so the method will skip this log file.

Solution

Due to multi-thread error common use of KVstore Listing, we make the cleanAppData method atomic, just like the addListing mothod, this allows threads to use KVstore Listing without data errors.

For example:

If Thread 1 gets the lock on the KVstore Listing before Thread 2, it will update the record of <appId, InfoData<appId>>, so Thread 2 can safely get the record and it will make isStale be false, so the cleanAppData method will not delete the record in KVstore Listing.

If Thread 2 gets the lock on the KVstore Listing before Thread 1, it will make isStale be true, and the cleanAppData method will delete the record in KVstore Listing. But it will be ok, beacuse Thread 1 will add the record of <appId, InfoData<appId>> in KVstore Listing again.

In all situations above, the record of KVstore Listing will not remove unexpectedly.

This is the method used to fix the problem in Spark v3.2.2 (https://github.com/apache/spark/pull/36424/files).
