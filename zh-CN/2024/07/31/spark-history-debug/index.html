Spark HistoryServer 介绍与故障排查

HistoryServer 介绍

HistoryServer 是 Spark 内置的一个 HTTP 服务，我们可以通过下面的命令启动它。HistoryServer 启动后，会开启两个线程分别用于解析和清理日志文件。

该命令会在 http://<server-url>:8080 上创建一个 Web 界面，用于展示尚未完成以及已经完成的应用与尝试（attempt）。

Spark 作业本身必须配置为记录事件，并将事件日志写入同一个共享的、可写的目录。

为了更好地介绍 Spark History Server，下面的流程图梳理了它的工作流程。

Spark History Server 依赖于 Spark 的事件（Event）体系。在一个 Spark 任务运行过程中，会产生各种 SparkListenerEvent，例如 ApplicationStart、StageCompleted 等。这些事件会被发送到 ListenerBus，并被已注册的监听器监听。其中，EventLoggingListener 会将这些事件以 JSON 日志文件的形式写入到诸如 HDFS 这样的文件系统中。

History Server 中的 FsHistoryProvider 会周期性地扫描日志存储路径，从日志文件中抽取概要信息（例如 Application_id、user、status、start_time、end_time、event_log_path），并维护一份数据列表。当用户访问 UI 时，系统会在该列表中查找对应的任务，如果存在，就读取相应的事件日志文件以展示详情。

问题

在一些情况下，使用 Spark 提交作业之后，在 Spark HistoryServer WebUI (v3.2.1) 上无法找到对应 appId 的日志信息。

因此，用户无法通过 HistoryServer（HS）来分析作业日志。在分析源码之后，发现问题的根因在于多线程环境下对某个数据结构的错误使用。

日志解析流程

HS 默认每 10 秒运行一次 checkForLogs 方法，以根据日志目录的当前内容来构建应用列表。每次执行时，该方法都会扫描指定的 logDir，并解析日志文件以更新其 KVStore listing。整个流程包含两个主要的方法：

  1. 分别更新 / 新增日志文件记录
  2. 移除过期的日志文件记录。

这些方法通过 submitLogProcessTask 在不同的线程中执行。

需要注意的是，日志文件记录 <'ABC', InfoData<'ABC'>> 不会再被添加回 KVStore Listing。因为在下一次 checkForLogs 运行时，由于 <'/tmp/ABC', InfoData<'/tmp/ABC'>> 这条记录的存在，表示 ABC 的日志文件无需重新加载，所以该方法会直接跳过这个日志文件。

解决方案

由于在多线程环境下对 KVStore Listing 的使用存在问题，我们把 cleanAppData 方法变为原子操作，就像 addListing 方法一样，从而让多个线程都可以安全地使用 KVStore Listing，而不会出现数据错误。

例如：

如果线程 1 先于线程 2 获取到 KVStore Listing 的锁，它会更新 <appId, InfoData<appId>> 这条记录，这样线程 2 就可以安全地读取到该记录，进而使 isStale 为 false，cleanAppData 方法也就不会再删除 KVStore Listing 中的记录。

如果线程 2 先于线程 1 获取到 KVStore Listing 的锁，它会使 isStale 为 true，cleanAppData 方法会删除 KVStore Listing 中的记录。但这没关系，因为线程 1 随后会再次向 KVStore Listing 中写入 <appId, InfoData<appId>> 这条记录。

在上述所有情况下，KVStore Listing 中的记录都不会被意外移除。

这就是 Spark v3.2.2 中用来修复该问题的方法（https://github.com/apache/spark/pull/36424/files）。
